"Build a productized 'Blog OS' system for a client that automates research, writing, CMS management, and website publishing of blog content end-to-end. The system should feel like a managed service/product line, not a custom one-off build. It must be designed so we, or agents operating on our behalf, can deliver it consistently as a sellable system.\n\n## Deliverable\nProduce the full implementation for an automated blog publishing suite called 'Blog OS'. This includes a Notion-backed editorial CMS, an AI research + drafting pipeline, an approval workflow, and a pipeline that publishes approved posts to a marketing website. The delivery should include code, integration hooks, runbooks, and a client-facing product spec with clear package tiers.\n\n## System Architecture\nBuild the following subsystems:\n\n### 1. Client/Product Setup\n- Create one or more Notion workspaces/databases for each client: a Posts DB, and optionally a Campaigns or Ideas DB.\n- Each Posts DB row tracks: title, excerpt, slug, category, tags, status, author, date, content blocks, SEO fields, publish URL, and workflow metadata.\n- Initialize with a schema that maps cleanly to the target website's data model.\n\n### 2. AI Research Pipeline\n- Accept a topic, keyword cluster, or competitor set as input.\n- Research phase must:\n  - Query web sources for related queries, top-ranking pages, and content gaps.\n  - Produce a structured brief: primary keyword, supporting keywords, audience intent, outline, internal linking candidates, and differentiation angle.\n- Store the brief in Notion before writing begins.\n\n### 3. AI Drafting Pipeline\n- Convert the approved brief into a full draft post in Notion.\n- Draft must include: title, excerpt, category, tags, author, date, and structured content blocks (text, headings, code if relevant, CTAs).\n- Enforce editorial guardrails: tone/brand voice, word count target, factual grounding from research, and citation/reference links.\n- Create a Notion row in draft status.\n\n### 4. Approval Workflow\n- Support human review directly inside Notion.\n- Statuses must include: idea → brief → draft → review → approved → scheduled → published → archived.\n- Reviewers can edit title, excerpt, content, status, and tags in Notion.\n- Changes should be auditable via Notion history; the system should be able to detect status transitions.\n\n### 5. Publish Pipeline\n- A sync process that watches the client's Posts DB for approved items and publishes them to the website.\n- If the site is Next.js, implement RSC/API routes that read Notion and render pages safely with ISR or SSG where appropriate.\n- If the site is WordPress, implement the REST/Gutenberg path.\n- Put each post at a clean URL derived from slug and ensure SEO metadata is written.\n\n### 6. Distribution Hooks\n- Optional but recommended:\n  - Create social snippets and threads from the finished post.\n  - Generate short-form asset prompts for video/audio versions.\n  - Queue distribution tasks for execution by a distribution agent.\n\n## Technical Requirements\n- Use Notion as the editorial source of truth via direct Notion REST API.\n- Provide a secure integration layer with proper token/env handling and least-privilege access.\n- Include content deduplication checks so similar topics or URLs are flagged before publishing.\n- Include a cron or event trigger that detects new Notion rows/status changes and runs the next stage automatically or notifies the responsible agent/human.\n- Log every stage in Notion as properties or child blocks for full traceability.\n\n## Interface & UX Expectations\n- Client view should be a simple Notion dashboard or light admin page that shows:\n  - upcoming queue\n  - in-review items\n  - recently published\n  - publishing health/errors\n- Support white-label output: any Notion workspace should be mountable without custom code changes if schema matches.\n\n## Sellable Packaging\n- Define 3 tiers:\n  - DIY Blueprint: templates, prompts, schema, and setup guide so the client runs it themselves.\n  - Managed System: we operate the pipeline, handle research, drafting, scheduling, publishing, and light distribution reporting.\n  - Enterprise OS: multi-author, multi-site, SEO automation, advanced analytics, and dedicated ops agent.\n- Each tier must map to clear technical implementation differences so pricing and delivery are enforceable.\n\n## Implementation Order\n1. Create the Notion schema and seed rows.\n2. Build research and drafting functions that write to Notion.\n3. Add approval/status management logic.\n4. Build the publishing integration for a specific website stack.\n5. Add monitoring/logging and distribution hooks.\n6. Package with runbooks, schemas, and client handoff materials.\n\n## Verification\nBefore delivery, verify:\n- A new topic can be entered, researched, drafted, approved, and published without manual file edits outside Notion and the configured website.\n- Status transitions are durable across sessions.\n- Published content is visible on the live site with correct SEO metadata.\n- Error states are handled cleanly: token expiry, rate limits, empty fields, orphaned rows, and duplicate URLs.\n"
